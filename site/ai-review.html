AI Schema Review

AI Schema Review sends your loaded schema-and-mapping document to a large language model via OpenRouter and returns a prioritised set of recommendations covering naming, structure, mapping coverage, cardinality, best practices, performance, and domain completeness. The tool infers the real-world domain your schema models and flags obvious gaps alongside structural issues.

Where to find it Tools panel → AI Schema Review

The last card in the Tools panel. Expand it, pick a model, and click Run AI Review. Results appear inline with a quality score, executive summary, and a categorised finding list.

What it produces Score + prioritised findings

A 0–100 quality score, a 2–3 sentence executive summary, and up to 15 findings — each with severity, category, description, path, and a concrete remediation step.

Models

Choose the right model for your workflow

Model Badge Context Best for
Gemini 2.0 Flash Fast 1 M tokens Routine checks during active schema editing. Returns results in seconds at minimal cost. Handles even the largest schemas in full-document mode.
Gemini 2.5 Pro Deep 1 M tokens Pre-production review or schemas with complex inheritance hierarchies. Uses extended reasoning to surface subtle anti-patterns and cross-cutting issues.
Claude Opus 4.5 Expert 200 K tokens Formal governance reviews where precise, actionable language matters. Uses a structured consultant-style prompt with explicit scoring arithmetic.

Each model receives a tailored system prompt — not a generic one. The prompts differ in framing, reasoning strategy, and output style to get the most out of each model's specific strengths.

Analysis categories

What gets checked

Category What the model looks for
Naming Consistent capitalisation conventions, duplicate names at the same hierarchy level, names with special characters or starting with digits.
Structure Hierarchy depth, correct use of abstractObject flags, inheritance wiring, orphaned branches with no children or mappings.
Mapping Unmapped object types, missing selectors or locators, fragile IQL expressions, attribute mappings with empty locators, coverage ratio.
Cardinality minimumCardinality > maximumCardinality violations, boolean attributes with maximumCardinality > 1 (logically invalid), required attributes on abstract types (unenforced).
Best Practice At least one label: true attribute per concrete type, description fields on top-level types, human-readable externalId values, statusSchema alignment.
Performance Object types with more than 25 attributes, hierarchies deeper than 5 levels, schemas with less than 50% mapping coverage that will silently skip most types on import.
Completeness The model first infers the real-world domain (e.g. IT asset management, HR, network infrastructure) from type and attribute names, then identifies entities or relationships clearly implied by the domain but absent from the schema. Only obvious, strongly-implied gaps are flagged — not wishlist additions.
Quality score

How the 0–100 score is calculated

Each model uses a severity-weighted deduction approach. The score starts at 100. Each error deducts 10–15 points; each warning deducts 3–5 points; info findings have minimal or no impact. The score never goes below 0.

85–100
Production-ready

Clean, complete, well-structured. Minor polish only.

70–84
Minor issues

Ready with small fixes. Address warnings before go-live.

50–69
Moderate issues

Meaningful improvements needed before production.

0–49
Significant problems

Structural or mapping issues likely to cause import failures.

Large schemas

Full-document vs summary mode

Before sending to the API, the tool estimates the token count (characters / 4). If the full JSON fits within the model's context budget (context window minus 20 K reserved for the response), it sends the complete document. Otherwise it sends a structural summary instead.

Full mode Complete JSON sent

All attribute values, selectors, IQL expressions, and locators are visible to the model. Produces the most accurate findings. Used by default for any schema that fits the context.

Summary mode Structural skeleton sent

Extracts type names, attribute names and types, mapping selector counts, and unmapped type annotations. Reduces input by ~20×. Used automatically when full JSON would exceed the context budget.

Gemini 2.0 Flash and Gemini 2.5 Pro both have a 1 M token context window, so full-document mode applies to virtually all real-world schemas. Claude Opus 4.5 has a 200 K token limit and may fall back to summary mode for very large schemas.

The result header shows which mode was used alongside the token estimate.

Setup

Configuration

Add your OpenRouter API key to .env.local in the project root:

OPENROUTER_API_KEY=sk-or-v1-...

Restart the dev server after adding the key. In production (Azure App Service), the deploy script reads the key from .env.local and stores it in Key Vault automatically on first deploy.

The key never reaches the browser

OPENROUTER_API_KEY is read exclusively by the POST /api/tools/llm-review server route. It is not prefixed with NEXT_PUBLIC_ and is never included in the client bundle.

API route

POST /api/tools/llm-review

FieldTypeDescription
documentAssetsImportDocumentRequired. The full schema-and-mapping payload.
model"gemini-flash" | "gemini-pro" | "claude-opus"Optional. Defaults to gemini-flash.

Response shape:

{
  "summary": "...",
  "score": 78,
  "recommendations": [
    {
      "severity": "warning",
      "category": "mapping",
      "title": "...",
      "description": "...",
      "path": "/schema/objectSchema/objectTypes/3",
      "suggestion": "..."
    }
  ],
  "model": "Gemini 2.0 Flash",
  "mode": "full",
  "inputTokenEst": 12400
}

The route has a 120-second timeout (export const maxDuration = 120) to accommodate large schemas on slower models.
